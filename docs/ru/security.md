# Безопасность

Этот документ описывает модель угроз, средства контроля, применяемые `gpg-keysmith`, и явные нецели. Это справочный материал для проверки уровня безопасности инструмента.

## Защищаемые активы

| Актив | Почему важен | Где живёт |
|---|---|---|
| Парольная фраза | Открывает приватный ключ; её утечка = утечка ключа | Только в памяти, подаётся в `gpg` через stdin |
| Приватный ключ | Позволяет подписывать коммиты/атаки от имени пользователя | Связка ключей GPG на диске (`~/.gnupg`), но `gpg-keysmith` экспортирует его только в память — никогда в файл и никогда в лог |
| GitHub PAT | Позволяет писать в репозитории, выставлять секреты, загружать ключи | Переменная окружения, имя которой задано в `config.github.token_env` (по умолчанию `GITHUB_TOKEN`), запасной вариант `GH_TOKEN` |

## Средства контроля

### Парольная фраза — только stdin, никогда аргумент CLI

Парольная фраза собирается через маскированное поле `survey.Password` и подаётся в `gpg` через `--passphrase-fd 0` (stdin) с `--pinentry-mode loopback`.

| Поверхность утечки | Как избегаем |
|---|---|
| `ps` / `/proc/cmdline` | Парольная фраза никогда не является аргументом CLI — в argv `exec.Command` только `gpg`, флаги и провалидированный key ID |
| Batch-файл | Директива `%no-protection` намеренно отсутствует; парольная фраза никогда не пишется в batch-файл, используемый `gpg --gen-key` |
| Логи | Парольная фраза никогда не печатается, не логируется и не возвращается из функции в значении, которое попадает в лог |
| История shell | Парольная фраза собирается через `survey.Password` (маскированный ввод), а не через флаг, который пользователь печатает в shell |

### Приватный ключ — только в памяти

Вывод `gpg --armor --export-secret-keys` перехватывается в Go-срез `[]byte` в памяти процесса. Он:

- никогда не пишется на диск (без временных файлов, без кэша);
- никогда не логируется;
- никогда не печатается в stdout;
- хранится в памяти между шагом `export` и шагом `github` (для загрузки секретов командой `github`), затем уничтожается при выходе процесса.

Структура `WizardState` несёт поля `Passphrase`, `PrivateKey` и `PubKeyArmor` с тегом `json:"-"`, поэтому они хранятся в памяти между шагами мастера, но никогда не сериализуются в файл `state.json`. Инвариант проверяется тестом `TestSaveStateOmitsSecrets`.

### GitHub PAT — только переменная окружения, без флага `--token`

PAT берётся только из переменных окружения:

1. `config.github.token_env` (по умолчанию `GITHUB_TOKEN`)
2. `GH_TOKEN` (запасной вариант)

Флаг `--token` был **удалён** при усилении безопасности (S1), потому что значение флага утекает через `ps` и `/proc/cmdline` на любой многопользовательской системе. Конфиг хранит только **имя переменной окружения**, в которой лежит токен, а не сам токен. Файл конфига пишется в режиме `0600`.

### Валидация идентификаторов — защита от инъекции

Каждый пользовательский идентификатор, который попадает в подпроцесс или URL, валидируется до того, как туда попадёт.

| Идентификатор | Валидатор | Что отвергает | Где используется |
|---|---|---|---|
| Key ID | `ValidateKeyID` | Не-hex символы, длина > 40 | `export`, `github`, `publish`, `wizard` |
| Fingerprint | hex-проверка | Не-hex символы | `publish` (URL сервера ключей), `status` (проверка на сервере) |
| owner/repo | `ValidateOwnerRepo` | Всё вне `^[A-Za-z0-9._-]+$` | `github`, `status`, `wizard` |

`ValidateOwnerRepo` вызывается в каждой публичной точке входа, принимающей owner и repo (`SetRepoSecret`, `SetGPGSecrets`, `CommitPublicKeyFile`, `ListRepoSecrets`). Как дополнительная мера, вызывающие стороны, интерполирующие owner/repo в пути URL, применяют `url.PathEscape`.

Поскольку `exec.Command` не вызывает shell, shell-инъекция невозможна, даже если бы валидатор что-то пропустил — провалидированная строка становится единственным элементом argv.

### Файл конфигурации — 0600, только имя token_env

| Свойство | Значение |
|---|---|
| Режим файла | `0600` (записывается `config.Save` / `config.Init`) |
| Режим родительского каталога | `0700` |
| Хранение токена | Только **имя** переменной окружения (`token_env: GITHUB_TOKEN`), не значение |
| Пустой `token_env` | `Save` отказывается писать — `ErrEmptyTokenEnv` — чтобы не дать случайно сохранить значение токена напрямую |

### Файл состояния — без секретов

Файл состояния мастера (`~/.config/gpg-keysmith/state.json`) хранит только имена шагов, key id, email и репозиторий. Он **никогда** не содержит парольную фразу и приватный ключ — поля `Passphrase`, `PrivateKey` и `PubKeyArmor` структуры `WizardState` несут тег `json:"-"`, что проверяется тестом `TestSaveStateOmitsSecrets`.

### Валидация имени секрета (секреты репозитория)

`SetRepoSecret` проверяет имя секрета на соответствие разрешённому GitHub набору символов (`^[A-Za-z_][A-Za-z0-9_]*$`) перед вызовом `gh secret set`. Это предотвращает интерполяцию некорректного имени в вызов `gh`.

## Модель угроз

| Угроза | Вектор | Средство контроля |
|---|---|---|
| Утечка парольной фразы через `ps` | Кто-то на том же хосте запускает `ps`, пока работает `keysmith generate` | Парольная фраза подаётся через stdin, никогда через аргумент CLI |
| Утечка парольной фразы через batch-файл | Batch-файл `gpg --gen-key` остаётся на диске | Парольная фраза никогда не пишется в batch-файл |
| Запись приватного ключа на диск | Вывод `gpg --armor --export-secret-keys` перенаправляется в файл | Вывод перехватывается в Go-срез `[]byte`, а не в файл |
| Приватный ключ в файле состояния | Мастер пишет состояние в `state.json` между шагами | `json:"-"` на `WizardState.PrivateKey`; проверяется `TestSaveStateOmitsSecrets` |
| Утечка PAT через `ps` | `--token ghp_...` виден в `ps` | Флаг `--token` удалён; PAT читается только из переменной окружения |
| Утечка PAT через файл конфига | Кто-то читает `~/.config/gpg-keysmith/config.yaml` | Конфиг хранит только имя переменной окружения; файл в режиме `0600` |
| Инъекция key ID в argv `gpg` | `--keyid "ABCD; rm -rf ~"` передаётся как один элемент argv | `ValidateKeyID` отвергает не-hex; `exec.Command` не вызывает shell |
| Инъекция owner/repo в URL GitHub REST | `--repo "owner/bar/../other"` интерполируется в `/repos/%s/%s/...` | `ValidateOwnerRepo` отвергает всё вне `^[A-Za-z0-9._-]+$`; `url.PathEscape` как дополнительная мера |
| Инъекция fingerprint в URL сервера ключей | Не-hex fingerprint интерполируется в `https://keys.openpgp.org/vks/vby/<fp>` | Fingerprint проверяется на hex перед построением URL |

## Нецели (что keysmith НЕ делает)

- **Нет поддержки HSM / аппаратных ключей.** Инструмент работает с программной связкой ключей `gpg`. Поддержка смарт-карт / YubiKey OpenPGP вне области.
- **Нет собственного сервера ключей.** Публикация только на существующие серверы (`keys.openpgp.org`, `keyserver.ubuntu.com`).
- **Нет SSH-ключей.** Только подписные ключи OpenPGP — `gpg.format=openpgp` — единственное значение, которое `git-config` выставляет.
- **Нет age / X.509 / других схем подписи.** Только OpenPGP.
- **Нет workflow ротации ключа.** `generate` создаёт новый ключ; он не замещает и не отзывает существующий. Используйте `gpg` напрямую для отзыва.
- **Нет мультитенантной изоляции ключей.** Инструмент однопользовательский; связка ключей GPG, с которой он работает, — пользовательская `~/.gnupg`.

## Сообщение о проблемах безопасности

Если вы считаете, что нашли проблему безопасности в `gpg-keysmith`, сообщите о ней ответственно, а не открывайте публичный issue:

1. **Не** открывайте публичный issue на GitHub.
2. Напишите мейнтейнеру на `korrnals@gmail.com` с описанием проблемы и шагами воспроизведения.
3. Если проблема связана с утечкой секретного материала, немедленно смените затронутый секрет (парольную фразу, PAT или ключ) перед сообщением — сам отчёт никогда не должен содержать живых секретов.

Отчёты подтверждаются в течение 7 дней. Исправление и раскрытие координируются с репортёром.
